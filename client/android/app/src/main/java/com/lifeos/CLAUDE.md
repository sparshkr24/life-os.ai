# CLAUDE.md — `client/android/app/src/main/java/com/lifeos/`

## Responsibility

All native Android (Kotlin) code. Three concerns: (1) bridge JS calls into native, (2) keep a foreground service alive 24/7, (3) collect from OS APIs and write events directly into the SQLite file Expo uses.

**Schema is owned by JS.** Kotlin only does `INSERT` / `SELECT`. If a column is missing, fail loudly.

## Files

| File | Purpose |
|---|---|
| `MainActivity.kt`, `MainApplication.kt` | Generated by `expo prebuild`. `MainApplication` registers `LifeOsBridgePackage`. `MainActivity` carries an extra intent-filter for `androidx.health.ACTION_SHOW_PERMISSIONS_RATIONALE`. |
| `LifeOsBridgePackage.kt` | Standard `ReactPackage` — registers `LifeOsBridgeModule`. |
| `LifeOsBridgeModule.kt` | All RN-callable methods. See bridge contract table below. |
| `LifeOsForegroundService.kt` | Sticky `START_STICKY` foreground service. 60s poll reads `UsageStatsManager`; every 5 min calls `HealthConnectCollector.pollOnce`. Registers ActivityRecognition transitions and Sleep segment updates on `onCreate`. **Heartbeat**: writes `schema_meta.service_heartbeat_ts` every poll for the watchdog. **Notification**: updates content text every poll (`Active · uptime Xm`) so the user can see liveness in the shade. **FG dedup**: on `MOVE_TO_FOREGROUND` for `pkg`, if the latest `app_fg` row for that pkg has `end_ts` within `DEDUP_GAP_MS` (90s), extends that row in place. **Implicit close**: when app B foregrounds, any open session for app A is closed at B's ts. **Noise filter**: known system shells (launchers, intent resolver, permission controller, system UI, IMEs) dropped via `NOISE_PKGS` + `NOISE_PREFIXES`. |
| `BootReceiver.kt` | `BOOT_COMPLETED` receiver — restarts the foreground service AND schedules `WatchdogAlarmReceiver` after reboot. |
| `NightlyAlarmReceiver.kt` | Daily 03:05 `AlarmManager` kicker. Re-schedules itself with `setAlarmClock` (falls back to `setAndAllowWhileIdle` if `SCHEDULE_EXACT_ALARM` denied). Just calls `startForegroundService` — the actual nightly brain runs in JS via `maybeRunNightlyWatchdog()` inside the next aggregator tick. |
| `WatchdogAlarmReceiver.kt` | 15-min reliability watchdog. Reads `schema_meta.service_heartbeat_ts`; if older than 3 min, calls `startForegroundService` AND raises a high-importance notification (`lifeos_alerts` channel) so the user knows even if the app is closed. Re-schedules itself one-shot via `setExactAndAllowWhileIdle`. Scheduled from `LifeOsForegroundService.onCreate` and `BootReceiver.onReceive`. |
| `AggregatorHeadlessTaskService.kt` | Drives the JS aggregator every 15 min when the app UI is killed. Extends `HeadlessJsTaskService`; boots a JS context to run `LifeOsAggregator`. Started from `LifeOsForegroundService.maybeKickAggregator()` on the 60s poll. Coordinates with the JS-side foreground driver via `schema_meta.last_aggregator_ts`. 4-min hard timeout. |
| `EventDb.kt` | Shared `object EventDb`. Three calls: `insert(ctx, kind, ts, payloadJson)`, `insertReturningId(...)`, `updatePayload(ctx, id, payloadJson)`. Opens `lifeos.db` in **rollback-journal mode (NOT WAL)** with `PRAGMA busy_timeout = 5000`. **Every payload runs through `PhoneState.stamp(...)` before insert.** |
| `PhoneState.kt` | Process-wide ambient state stamped onto every event. Holds `placeId`, `batteryPct`, `isCharging`, `networkType`, plus a public `appContext` weak handle used by `PlaceDetector` to write events from the main-looper handler after the originating receiver returned. Updated by passive listeners (battery broadcast, `ConnectivityManager.NetworkCallback`). **No fresh GPS calls.** `placeId` updated only by `GeofenceReceiver`. `stamp(payloadJson)` inserts `,"_ctx":{...}` before the closing `}`. Idempotent. |
| `ActivityTransitionReceiver.kt` | Receives `ActivityTransitionResult` broadcasts. Writes `activity` events with `{activity, direction:"enter"\|"exit"}`. Also forwards every transition to `PlaceDetector.onActivityTransition` so the dwell state machine can start/stop. |
| `SleepReceiver.kt` | Handles `SleepSegmentEvent` (writes segment with `start_ts/end_ts/duration_ms/status`) and `SleepClassifyEvent` (writes classify with `confidence/motion/light`). Both as `sleep` event kind. |
| `GeofenceReceiver.kt` | Receives `GeofencingEvent`. For each triggering fence: updates `PhoneState.placeId`, calls `PlaceDetector.onGeofenceEnter` (so we don't double-detect a known place as a fresh dwell). Reads `places.kind` per call; if `kind='ignored'`, suppresses the `geo_enter`/`geo_exit` write. All places are registered (manual/auto/ignored) — the kind only gates the event-write. |
| `PlaceDetector.kt` | Singleton dwell state machine. Tracks one open dwell at a time on a main-looper handler. `still+enter` (and not already inside a known fence) starts a dwell → at 8 min captures a single FusedLocation fix (`PRIORITY_BALANCED_POWER_ACCURACY`) → at 15 min writes a `place_visit` event with `status='pending_geocode'`. Any other activity-transition exit closes the open row by mutating `departure_ts`/`duration_ms`. Service shutdown cancels pending callbacks. |
| `GeofenceManager.kt` | `object GeofenceManager` with `setGeofences(ctx, places, onDone)` and `removeAll(ctx, onDone)`. Removes existing fences before re-adding. `INITIAL_TRIGGER_ENTER`, `NEVER_EXPIRE`, ENTER|EXIT. |
| `HealthConnectCollector.kt` | `isAvailable(ctx)` + `pollOnce(ctx, sinceMs)`. Reads `StepsRecord` + `HeartRateRecord` since last cursor. HR condensed to `{min, max, avg, n, start_ts, end_ts}`. Silent noop when HC not installed. |
| `StepCounterCollector.kt` | Hardware-sensor fallback for steps. Registered from FG-service `onCreate` when `ACTIVITY_RECOGNITION` is granted. `flushIfDue()` called from the 60s poll. Emits one `steps` event per 5-minute window, matching the Health Connect payload shape so `aggSteps` SQL is source-agnostic. Handles device-reboot baseline reset. |

## Bridge Methods (JS ↔ Kotlin contract)

| Method | Returns | Effect |
|---|---|---|
| `startService()` | `Promise<void>` | `startForegroundService(LifeOsForegroundService)` |
| `stopService()` | `Promise<void>` | `stopService(LifeOsForegroundService)` |
| `hasUsageAccess()` | `Promise<boolean>` | `AppOpsManager.OPSTR_GET_USAGE_STATS` check |
| `openUsageAccessSettings()` | `Promise<void>` | Opens `ACTION_USAGE_ACCESS_SETTINGS` |
| `getStats()` | `Promise<{eventsLastHour, totalEvents, lastInsertTs, dbExists}>` | Read-only counts from `events` |
| `hasActivityRecognitionPermission()` | `Promise<boolean>` | `ContextCompat.checkSelfPermission(ACTIVITY_RECOGNITION)` |
| `requestActivityRecognitionPermission()` | `Promise<void>` | Activity-level `requestPermissions` |
| `hasLocationPermissions()` | `Promise<{fine, background}>` | Both checked separately |
| `requestForegroundLocation()` / `requestBackgroundLocation()` | `Promise<void>` | Sequenced fg-first |
| `setGeofences(places[])` | `Promise<number>` | Removes existing, re-adds; returns count |
| `removeAllGeofences()` | `Promise<void>` | Drops all fences for the shared PendingIntent |
| `getCurrentLocation()` | `Promise<{lat, lng, accuracyM, ts}>` | One-shot fix via `FusedLocationProvider.getCurrentLocation`. Rejects with `permission_denied` or `no_location`. |
| `isHealthConnectAvailable()` | `Promise<boolean>` | `HealthConnectClient.getSdkStatus` |
| `openHealthConnect()` | `Promise<void>` | `HEALTH_HOME_SETTINGS` or Play Store fallback |

<service android:name=".LifeOsForegroundService" android:exported="false"

  <intent-filter><action android:name="android.intent.action.BOOT_COMPLETED"/></intent-filter>

## First-launch call flow

```
App.tsx → LifeOsBridge.startService()
  → LifeOsBridgeModule.startService
    → startForegroundService(LifeOsForegroundService)
      → onCreate(): notification channel + PhoneState.init() + poll loop (60s)
        → UsageStatsManager.queryEvents(lastPollMs, now)
        → MOVE_TO_FOREGROUND same pkg within 90s → extend existing session
        → MOVE_TO_FOREGROUND new pkg → INSERT new app_fg row
        → MOVE_TO_BACKGROUND → UPDATE last open row with end_ts
```

## After reboot

```
ACTION_BOOT_COMPLETED → BootReceiver.onReceive
  → startForegroundService(LifeOsForegroundService)
  → WatchdogAlarmReceiver.schedule(ctx)
    (same onCreate path as above)
```

## Reliability: heartbeat + watchdog

```
LifeOsForegroundService.onCreate
  → writeMeta("service_start_ts", now)
  → writeMeta("service_heartbeat_ts", now)
  → WatchdogAlarmReceiver.schedule(ctx)  // T+15min
  → pollTask every 60s:
     — collectUsage()
     — maybeKickAggregator()
     — writeMeta("service_heartbeat_ts", now)
     — updateForegroundNotification(now)

WatchdogAlarmReceiver fires (every ~15min):
  → read schema_meta.service_heartbeat_ts
  → stale (>3 min)?
     yes → startForegroundService + show IMPORTANCE_HIGH notification "Collector stopped"
     no  → cancel any prior alert notification
  → schedule(ctx)  // chain next tick
```

The service heartbeat is what the JS Today screen pill reads (via `schema_meta.service_heartbeat_ts`) to decide live/idle/stalled — not `events.ts`, since events only fire when the user opens an app.
