# CLAUDE.md — `client/android/app/src/main/java/com/lifeos/`

## Responsibility

All native Android (Kotlin) code. Three concerns: (1) bridge JS calls into native, (2) keep a foreground service alive, (3) collect from `UsageStatsManager` and write rows directly into the SQLite file Expo uses.

**Schema is owned by JS.** Kotlin only does `INSERT` / `SELECT`. If a column is missing, fail loudly.

## Files

| File | Purpose |
|---|---|
| `MainActivity.kt`, `MainApplication.kt` | Generated by `expo prebuild`. `MainApplication` registers `LifeOsBridgePackage` in `getPackages()`. `MainActivity` carries an extra intent-filter for `androidx.health.ACTION_SHOW_PERMISSIONS_RATIONALE` so Health Connect can deep-link back. |
| `LifeOsBridgePackage.kt` | Standard `ReactPackage` — registers `LifeOsBridgeModule`. |
| `LifeOsBridgeModule.kt` | All RN-callable methods: usage (`startService`, `stopService`, `hasUsageAccess`, `openUsageAccessSettings`, `getStats`), Stage 3b (`hasActivityRecognitionPermission`, `requestActivityRecognitionPermission`), Stage 3c (`hasLocationPermissions`, `requestForegroundLocation`, `requestBackgroundLocation`, `setGeofences`, `removeAllGeofences`), Stage 3d (`isHealthConnectAvailable`, `openHealthConnect`), **v7 (`getCurrentLocation` — one-shot FusedLocation fix)**. One companion holds `TAG, REQ_AR=1101, REQ_LOC_FG=1102, REQ_LOC_BG=1103`. |
| `LifeOsForegroundService.kt` | Sticky `START_STICKY` foreground service with persistent silent notification. 60 s poll loop reads `UsageStatsManager` (Stage 3a) **and** every 5 min calls `HealthConnectCollector.pollOnce` (Stage 3d). On `onCreate` it also registers Activity Recognition transitions (IN_VEHICLE/ON_BICYCLE/ON_FOOT/RUNNING/STILL/WALKING) and Sleep segment updates via PendingIntents to the receivers below. **FG dedup**: on a `MOVE_TO_FOREGROUND` for `pkg`, if the latest `app_fg` row for that pkg has end_ts within `DEDUP_GAP_MS` (90 s) of the new ts, that row's end_ts is extended in place — collapses the RESUMED/PAUSED storm Android emits during sub-activity nav (e.g. WhatsApp scrolling) into a single session. **Implicit close on switch**: when RESUMED fires for app B, any still-open row for the previously foregrounded pkg A is closed at B's ts. This is the primary close path — `ACTIVITY_PAUSED` (type 2) is unreliable on swipe-from-recents and screen-off, while `ACTIVITY_STOPPED` (type 23) is now also accepted as a close trigger. `closeOpenSessionMs` no longer requires `start_ts == end_ts`; it simply advances `end_ts` whenever the new value moves forward (capped by `MAX_SESSION_MS = 12 h`). Per-poll log line `[UsageStats] pkg=X computed_duration=Ymin raw_events=Z` lets you sanity-check totals from logcat. **Noise filter**: known system shells (launchers, intent resolver, permission controller, system UI, screenshot/smartshot, IMEs) are dropped via `NOISE_PKGS` + `NOISE_PREFIXES` before any insert. |
| `BootReceiver.kt` | `BOOT_COMPLETED` receiver — restarts the foreground service after reboot. |
| `NightlyAlarmReceiver.kt` | Stage 8. Daily 03:05 `AlarmManager` kicker. Re-schedules itself with `setAlarmClock` (or `setAndAllowWhileIdle` if `SCHEDULE_EXACT_ALARM` is denied on Android 12+). Action `com.lifeos.NIGHTLY_KICK`. On fire it just calls `startForegroundService(LifeOsForegroundService)` — the actual nightly Sonnet rebuild runs in JS via `maybeRunNightlyWatchdog()` inside the next 15-min aggregator tick. `NightlyAlarmReceiver.schedule(ctx)` is called from `LifeOsForegroundService.onCreate` (idempotent). |
| `AggregatorHeadlessTaskService.kt` | Drives the JS aggregator every 15 min when the app UI is killed. Extends RN `HeadlessJsTaskService`; boots a JS context just long enough to run the `LifeOsAggregator` task (registered in [client/index.ts](../../../../../../index.ts)), which calls `runAggregatorTick()`. Started from `LifeOsForegroundService.maybeKickAggregator()` on the same 60-s poll loop that polls UsageStats. Coordination with the JS-side foreground driver via `schema_meta.last_aggregator_ts` — whichever ran more recently wins, no double-runs. Replaces the unreliable `expo-background-fetch` path on aggressive OEMs (ColorOS / RealmeUI / OneUI / MIUI). 4-min hard timeout. |
| `EventDb.kt` | Shared `object EventDb`. Three calls: `insert(ctx, kind, ts, payloadJson)` (fire-and-forget), `insertReturningId(...)` (returns row id; used by ongoing-notif tracker), `updatePayload(ctx, id, payloadJson)` (UPDATE existing row). All three open `<filesDir>/SQLite/lifeos.db` in **rollback-journal mode (NOT WAL)** with `PRAGMA busy_timeout = 5000`, write, close. WAL is incompatible with the JS-side expo-sqlite engine sharing the same file — the two libraries produce divergent `-shm` shared-memory layouts that corrupt page checksums. **Every payload is run through `PhoneState.stamp(...)` before insert** so `_ctx` lands on every row at the source. |
| `PhoneState.kt` (v3 / Stage C) | Process-wide ambient state stamped onto every event. `init(ctx)` is called from `LifeOsForegroundService.onCreate` and registers passive listeners only — battery via `ACTION_BATTERY_CHANGED` sticky broadcast, network via `ConnectivityManager.NetworkCallback`, audio polled at stamp time via `AudioManager.getDevices`. **No fresh GPS calls** — `placeId` is updated only by `GeofenceReceiver` on enter/exit. `stamp(payloadJson)` is a string-level merge that inserts `,"_ctx":{place_id?,batt?,charging?,net?,audio?}` before the closing `}`. Idempotent (skips if `_ctx` already present). |
| `ActivityTransitionReceiver.kt` | Receives `ActivityTransitionResult` broadcasts. Writes `activity` events with `{activity, direction:"enter"|"exit", source}`. Action: `com.lifeos.ACTIVITY_TRANSITIONS`. |
| `SleepReceiver.kt` | Handles both `SleepSegmentEvent` (writes `{kind:"segment", start_ts, end_ts, duration_ms, status}`) and `SleepClassifyEvent` (writes `{kind:"classify", ts, confidence, motion, light}`). Both as `sleep` event kind. Action: `com.lifeos.SLEEP_EVENTS`. |
| `GeofenceReceiver.kt` | Receives `GeofencingEvent`, distinguishes ENTER/EXIT, writes `geo_enter` / `geo_exit` rows with `{place_id (= requestId), lat, lng, source}`. **Also mirrors transitions into `PhoneState.placeId`** so subsequent events get stamped with the user's current place. Action: `com.lifeos.GEOFENCE_TRANSITIONS`. |
| `GeofenceManager.kt` | `object GeofenceManager` with `Place(id, lat, lng, radiusM)`, `setGeofences(ctx, places, onDone)`, `removeAll(ctx, onDone)`. Removes existing fences before re-adding; uses `INITIAL_TRIGGER_ENTER`, `NEVER_EXPIRE`, ENTER\|EXIT transitions. |
| `LifeOsNotificationListener.kt` | **Removed (post-v1.0.2).** The notification collector wrote pkg/category/ongoing only — no content — and produced no irreplaceable signal vs `app_fg`/`screen_on`/`activity`/`geo_*`. Permission, manifest entry, bridge methods, and `notif` event kind were all removed together. |
| `HealthConnectCollector.kt` | `object HealthConnectCollector` with `isAvailable(ctx)` and `pollOnce(ctx, sinceMs): Long`. Uses `runBlocking` to read `StepsRecord` + `HeartRateRecord` since the last cursor; HR samples are condensed to `{min, max, avg, n, start_ts, end_ts}` to keep payloads small. Silent noop on devices where HC isn't installed/whitelisted (iQOO/Vivo/Honor/etc.) — the step-counter sensor below is the fallback. |
| `StepCounterCollector.kt` | Hardware-sensor fallback for steps (`Sensor.TYPE_STEP_COUNTER`). Registered from FG-service `onCreate` when `ACTIVITY_RECOGNITION` is granted; the kernel batches callbacks so battery cost is negligible. `flushIfDue()` is called from the 60 s poll Runnable; emits one `steps` event per 5-minute window with shape `{count, start_ts, end_ts, source: "sensor_step_counter"}` matching the Health-Connect payload so `aggSteps` SQL doesn't care which source produced the row. Handles device-reboot baseline reset by detecting cumulative-counter regression. Works fully offline; no Google Play Services required. |

## Bridge methods (JS ↔ Kotlin contract)

| Method | Returns | Effect |
|---|---|---|
| `startService()` | `Promise<void>` | `startForegroundService(LifeOsForegroundService)` |
| `stopService()` | `Promise<void>` | `stopService(LifeOsForegroundService)` — used by Today’s Repair button before wiping the DB so Kotlin doesn’t race the rebuild |
| `hasUsageAccess()` | `Promise<boolean>` | `AppOpsManager.OPSTR_GET_USAGE_STATS` for our UID |
| `openUsageAccessSettings()` | `Promise<void>` | Opens `Settings.ACTION_USAGE_ACCESS_SETTINGS` deep-linked to our package |
| `getStats()` | `Promise<{eventsLastHour, totalEvents, lastInsertTs, dbExists}>` | Read-only counts from `events` |
| `hasActivityRecognitionPermission()` | `Promise<boolean>` | `ContextCompat.checkSelfPermission(ACTIVITY_RECOGNITION)` |
| `requestActivityRecognitionPermission()` | `Promise<void>` | Activity-level `requestPermissions` |
| `hasLocationPermissions()` | `Promise<{fine, background}>` | Both checked separately |
| `requestForegroundLocation()` / `requestBackgroundLocation()` | `Promise<void>` | Sequenced — fg first, then bg |
| `setGeofences(places: Array<{id, lat, lng, radiusM}>)` | `Promise<number>` | Removes existing then re-adds; returns count installed |
| `removeAllGeofences()` | `Promise<void>` | Drops all fences for the shared PendingIntent |
| `getCurrentLocation()` | `Promise<{lat, lng, accuracyM, ts}>` | **v7.** One-shot fix via `LocationServices.getFusedLocationProviderClient(ctx).getCurrentLocation(PRIORITY_HIGH_ACCURACY, …)` with `.lastLocation` fallback. Rejects with `permission_denied` when `ACCESS_FINE_LOCATION` is not granted, `no_location` when both providers return null. Used by `Settings → Places` (capture current point) and the proactive engine (`brain/proactive.ts` long_dwell_unknown trigger). |
| `isHealthConnectAvailable()` | `Promise<boolean>` | `HealthConnectClient.getSdkStatus(...) == SDK_AVAILABLE` |
| `openHealthConnect()` | `Promise<void>` | `android.health.connect.action.HEALTH_HOME_SETTINGS`, falls back to Play Store for `com.google.android.apps.healthdata` |

## Call flow examples

### 1. First launch — grant access + start service

```
App.tsx onPress "1. Grant Usage Access"
  → LifeOsBridge.openUsageAccessSettings()        // bridge/lifeOsBridge.ts
    → LifeOsBridgeModule.openUsageAccessSettings (Kotlin)
      → startActivity(ACTION_USAGE_ACCESS_SETTINGS)  // user toggles in Settings UI
  → user returns to app, taps "2. Start Service"
  → LifeOsBridge.startService()
    → LifeOsBridgeModule.startService
      → ContextCompat.startForegroundService(LifeOsForegroundService)
        → onCreate() creates notif channel + ongoing notification
        → handler.post(pollTask)  every 60s:
            → UsageStatsManager.queryEvents(lastPollMs, now)
            → for MOVE_TO_FOREGROUND with same pkg as last app_fg row, end_ts within 90s:
                → UPDATE that row's payload (extend session) — no new row inserted
            → for MOVE_TO_FOREGROUND otherwise:
                → INSERT new row with payload {pkg, start_ts, end_ts, duration_ms, source}
            → for MOVE_TO_BACKGROUND: UPDATE last open app_fg row's payload to stamp end_ts
```

### 2. After reboot

```
Android: ACTION_BOOT_COMPLETED broadcast
  → BootReceiver.onReceive
    → ctx.startForegroundService(LifeOsForegroundService)
```

## Where the SQLite file lives

`expo-sqlite` opens `<filesDir>/SQLite/lifeos.db` on Android. Kotlin opens the **same path** so JS and native share one DB. JS owns `CREATE TABLE` and runs migrations on app cold start. Kotlin must wait until the file exists (the service no-ops with a warn log if `lifeos.db` is missing — first app launch will create it).

## Hard rules

1. **No DDL from Kotlin.** Only `INSERT` / `SELECT`. If you need a new column, change `client/src/db/schema.ts`, bump `SCHEMA_VERSION`, restart app.
2. **Schema columns Kotlin INSERTs touch:** `events(ts, kind, payload)`. That's it. Anything else (source, app category, etc.) goes inside `payload` JSON. If you change column names in `schema.ts`, grep this folder for the old name in the same commit.
3. **No business logic.** Kotlin = "OS event in → row out". Rules, nudges, todos all live in JS or backend.
4. **Single bridge module.** Don't split into per-collector modules.
5. **Simplicity rule (root CLAUDE.md #1).** Prefer one-file collectors over abstractions. The whole `LifeOsForegroundService.kt` is ~120 lines. Keep it that way.
