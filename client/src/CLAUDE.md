# CLAUDE.md — `client/src/`

## Responsibility

All TypeScript application code for the Expo app. UI shell lives in
`client/App.tsx`; per-tab screens are in `screens/`. This folder owns:
local SQLite (`db/`), the native-bridge wrapper (`bridge/`), read-only repos
(`repos/`), secure-store helpers (`secure/`), and screen components (`screens/`).

Stage 4 lands schema v2 + observability tabs. Stage 5 lands the 15-min
aggregator pipeline (`aggregator/`). Stage 6 lands the rule engine + 3-level
local notifications (`rules/`). Brain and chat wiring arrive in later stages.

| `db/repair.ts` | `attemptRepair()` — best-effort recovery when `SQLITE_CORRUPT` surfaces mid-session. Algorithm: open a fresh read-only connection, page-scan every table from `sqlite_master` (LIMIT/OFFSET, skip up to 4 consecutive bad pages, advance offset on error), wipe the file via `SQLite.deleteDatabaseAsync`, run `migrate()` on the fresh file, then re-INSERT each rescued row with `INSERT OR IGNORE` (skipping `schema_meta`'s version row so future migrations still fire). Returns `RepairReport` with per-table `rescued`/`failed` counts. UI must call `LifeOsBridge.stopService()` before invoking and `startService()` after — Today's `RepairButton` does this. No `expo-file-system` dependency. |
| `db/index.ts` | DB lifecycle: `getDb`, `withDb`, `migrate`, `describeDb`, `reopenDb`, `purgeShortAppFg`, plus internal `addColumnIfMissing` for additive columns. Single open connection (`useNewConnection: true` to bypass the native zombie cache). **Journal mode = DELETE (not WAL):** the same `lifeos.db` file is opened by both expo-sqlite (JS) and Kotlin's `android.database.sqlite.SQLiteDatabase`; their WAL `-shm` shared-memory layouts disagree → page-checksum drift → `database disk image is malformed`. Rollback journal coordinates the two engines via OS file locks; `PRAGMA busy_timeout = 5000` on both sides absorbs momentary contention. Also exports `isDbCorrupt()` / `clearDbCorrupt()` — `withDb` flips the corrupt flag when it catches `SQLITE_CORRUPT` so `screens/Today.tsx` can render a Repair banner. `withDb(fn)` is the canonical entry point for queries — auto-recovers from stale-SharedRef NPEs by reopening once. `migrate()` runs `purgeShortAppFg(db, 1000)` on every boot, then unconditionally backfills v2-retroactive `nudges_log` columns (`source/rule_id/llm_call_id/reasoning/level`) and v3 columns (`productivity_score`, `nudges_log.{next_day_score,baseline_score,score_delta,user_helpful}`). The unconditional run is critical — users whose DB was created before `nudges_log` got those columns will hit `no such column: level` when smart-nudge inserts; the migration heals them on next boot. |
| `ingest/cleanup.ts` | `cleanupRawEvents()` — the JS-side ingestion pipeline. Single source of truth for post-write event cleanup. Runs at the start of every aggregator tick (before rollups). Three rules today: (1) drop `app_fg` rows whose `pkg` is in a noise denylist (`android`, launchers, settings, systemui, etc.) catching what slipped past Kotlin's write-time filter; (2) merge same-pkg `app_fg` sessions within 90s of each other in the last 24h (covers Kotlin service restarts that reset the in-memory dedup); (3) `purgeShortAppFg(1000)`. Idempotent, bounded. Also exports `WAKE_NOISE_PKGS` (launchers + lock-screen + alarm clocks + dialers) consumed by `aggWakeFirstApp`. **Add new cleanup rules here, not in collectors.** |
| `aggregator/silence.ts` | `classifySilences(db, date, tz)` — finds gaps ≥60 min in the active-event stream and writes `events.kind='inferred_activity'` with `label` ∈ {sleep_or_rest, focused_work, workout, unknown}. Uses last `geo_enter`/`geo_exit` for place context. Idempotent per day; preserves rows with `user_confirmed: true`. Called from the Stage-5 aggregator tick for today + yesterday. |
| `aggregator/time.ts` | Shared time helpers: `localHour`, `localDateStr`, `localMonthStr`, `localDayStartMs` (±14h IANA-tz search window), `prevDate`, `nextDate`, `deviceTz`. Hermes `Intl.DateTimeFormat` based — no Luxon. |
| `aggregator/rollup.ts` | `rebuildDailyRollup(db, date, tz)` — single-responsibility daily-rollup orchestrator. UPSERTs `daily_rollup`. Owns the entire pipeline: `aggApps` (joins `app_categories`), `sumByCategory`, `aggByHour` (per-hour walk), `aggLateNight` (21:00–26:00 overlap), `aggSleep` (longest segment in [−12h, +14h]), `aggWakeFirstApp`, `aggPlaces` (geo_enter/exit walk + carry from prior day), `aggTransitions`, `aggSteps`, `aggActiveMinutes`, `aggTodos`, `aggNudges`, `aggSilences`. Exports `DailyRollupData`, `AppAgg`, `SleepAgg`, `prevDate`, `nextDate`. Exceeds 400 LOC by design — see hard rule #12. |
| `aggregator/monthlyFold.ts` | `foldMonth(db, month)` — rolls all `daily_rollup` rows for `'YYYY-MM'` into a `monthly_rollup` row (top apps, sleep p50/p90, place hours, totals, avg productivity score). Idempotent. |
| `aggregator/index.ts` | `runAggregatorTick()` — the public entry point. Per tick: `cleanupRawEvents()` (ingest pipeline) → `classifySilences(today,yest)` → `rebuildDailyRollup(today,yest)` → `computeProductivityScore(today,yest)` → `maybeRebuildPredictiveInsights(today)` (90 min throttle) → once-per-day `foldMonth(prevMonth)` (gated via `schema_meta.last_monthly_fold_date`) → `runRulesOnceFromBackground()` → `maybeRunNightlyWatchdog()` (Stage 8 + 14, gated by 03:00 local + 20 h cooldown). **Stage 14 removed the per-tick smart-nudge call** — nudge rules are now curated nightly. Records `last_aggregator_ts`. Returns `TickReport`. |
| `rules/notify.ts` | `expo-notifications` wrapper. | `rules/engine.ts` | `evaluateRules({tz?})` — pure rule evaluator. Helpers compute today's app/category minutes (incl. open sessions), today's wake_ts (longest sleep segment), and current place label. |

| `brain/productivityScore.ts` | `computeProductivityScore(db, date)` — pure SQL UPDATE. 5-component weighted score in [0,1] (sleep 30, focus 25, wake 15, move 15, nudge 15). No LLM. Called by aggregator after every `daily_rollup` rebuild. |
| `brain/predictiveInsights.ts` | `maybeRebuildPredictiveInsights(db, date)` — pure-RAG (one embed call, no generation). | `brain/verifiedFacts.ts` | `buildVerifiedFacts(db)` — produces the `VERIFIED_FACTS` block for the nightly Sonnet call. Currently emits `low_phone_night` correlation when ≥5 days exist on each side; null otherwise. Add new correlations as new fns and append to the array. |

| `brain/nightly.ts` | **v3 Phase E + Stage 14 (three-pass, raw-events-aware).** `runNightlyRebuild()` + `maybeRunNightlyWatchdog()` + `lastNightlyTs()`. Pre-LLM idempotent SQL: finalise yesterday's `productivity_score` + 7-day `computeNudgeEffectiveness`. Then **three sequential tool-calling sessions** sharing one `runToolLoop` helper: **(1) `runMemoryPass(yesterday)`** — scope `'nightly_memory'` (max `MEMORY_TOOL_LOOPS=8`). Inputs: yesterday's FULL raw event timeline via `loadRawEventsForDate` (one event per line, `_ctx`-stamped, capped at `MAX_EVENTS_FOR_MEMORY=2000`), yesterday's daily_rollup, prior profile, unverified predictions, un-enriched app_category count. The model extracts new memories (`create_memory`), verifies predictions (`verify_memory`), reinforces/contradicts/archives, consolidates clusters (`consolidate_memories`), enriches `app_categories` (`set_app_category`). Side effects ARE the output — no JSON parsing. **After the tool loop succeeds, runs `runMemoryMaintenance()`** (Stage 15/16) — a deterministic SQL sweep that soft-archives failed predictions / consistently disproven memories / sub-floor confidence rows / aged-out consolidation children. Errors there log but don't fail the pass. **(2) `runProfilePass(yesterday)`** — scope `'nightly_profile'` (read-only, max `PROFILE_TOOL_LOOPS=4`). Inputs: prior profile, last 30 daily_rollup, last 3 monthly_rollup, `buildVerifiedFacts`, top-25 memories by `|impact|·confidence`. Final assistant message is the new `behavior_profile` JSON (validate fences + sentinel keys → UPSERT id=1). **(3) `runNudgePass(yesterday)`** — scope `'nightly_nudge'` (max `NUDGE_TOOL_LOOPS=6`). Inputs: just-rebuilt profile, all `source='llm'` rules, top-30 actionable memories (`|impact|≥0.15` AND `confidence≥0.5`, types causal/habit/prediction), 14-day `nudges_log` aggregate (fired/acted/dismissed/helpful_up/down). Refines existing LLM rules via `get_rule_effectiveness` → `update_rule`/`disable_rule`, then creates ≤4 new rules grounded in memories (`create_rule` rejects `predicted_impact_score < 0.15`). Side effects ARE the output. Pass cascade: cap-skip in any pass blocks the next; profile-failure also blocks nudge pass; `markNightlyComplete` runs if any pass produced output. Watchdog: local hour ≥ 3 AND >20h since last success. **Hard rules:** raw events go to the memory pass ONLY; the LLM may only mutate derived rows (memory feedback cols + app_categories + `source='llm'` rules) — never event payload/ts/kind, never memory summary/cause/effect/embedding, never user/seed rules. |
| `brain/rawEvents.ts` | **v3 Phase E.** `loadRawEventsForDate(date) → RawEventTimeline`. Loads every `events` row whose `ts` falls in the local day, sorted ascending. Each row is formatted as one line `HH:MM:SS [kind] <payload-json>` (payload still carries the `_ctx` ambient block stamped at insert time). Volume cap `MAX_EVENTS_FOR_MEMORY=2000`; over the cap, drops the lowest-signal `app_fg` rows (shortest durations under `APP_FG_DROP_THRESHOLD_MS=30s`) first, only trims oldest events as a last-resort guard. Returns `{date, totalEvents, emittedEvents, droppedAppFg, truncated, lines}`. Used exclusively by `runMemoryPass`. |

## Schema bump policy

When `SCHEMA_VERSION` changes:
1. Bump in `db/schema.ts`.
2. Update affected row types in same file.
3. `grep` Kotlin for any column you renamed/dropped (`client/android/app/src/main/java/com/lifeos`).
4. On phone: `adb uninstall com.lifeos` then reinstall — wipes the v(N-1) DB.

## LLM cost cap

Hard wall: $0.30/day default. Every LLM call (Stages 7–9) **must**:
1. `SELECT SUM(cost_usd) FROM llm_calls WHERE ts >= start_of_day` first.
2. Compare against `loadSnapshot().dailyCapUsd`.
3. Refuse and write a row with `ok=0, error='cap_exceeded'` if over.
